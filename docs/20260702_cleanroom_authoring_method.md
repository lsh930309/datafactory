# 2026-07-02 클린룸 문서 제작 방식 고정 메모

## 결론
- non-KIE/보고서형 문서는 공통 렌더러로 일괄 생성하지 않는다.
- 문서 1종당 실제 샘플을 시각적으로 분석하고, 해당 문서 전용 렌더러/디자인 언어로 제작한다.
- `컴플라이언스 점검보고서(RPT-08)` 대표본은 이 방식의 기준 샘플로 workbench에 적재했다.

## 성공한 방식
1. 원본 샘플에서 페이지 유형을 먼저 분해한다.
   - 표지, letter, contents, 조직도/규정, 운영실적, 교육표, 설문차트, 계획표 등.
2. 원본의 로고/문장/사진/스크린샷은 복제하지 않는다.
3. 가상 회사/가상 그래픽/가상 차트/가상 표를 새로 만든다.
4. 공통 표 템플릿을 반복하지 않고, 페이지 유형별 레이아웃을 다르게 만든다.
5. PDF 본문에는 합성/라이선스/검수 같은 내부 메타를 넣지 않는다.
6. 검수 메타는 manifest, QA notes, progress 문서에만 남긴다.

## 다음 문서 적용 원칙
- 한 번에 여러 문서를 만들지 않는다.
- 1종씩 `시각 분석 -> 전용 디자인 -> 렌더링 -> PDF 재렌더링 검수 -> workbench 적재` 순서로 처리한다.
- 이 방식으로도 품질이 나오지 않는 문서는 사용권 확보 샘플 기반 텍스트 치환 렌더링으로 전환한다.

## 2026-07-01 방식 보강
- `감사의견서(FIN-11)` A/B 비교 결과, A(Pillow 직접 드로잉)가 현재 목적에 더 적합하다고 판단했다.
- B(웹 조판 접목) 방식은 산출물 스타일과 프로젝트 목적의 적합도가 낮아 폐기한다.
- 이후 cleanroom 문서 제작 기본 원칙은 `문서별 시각 분석 + Pillow 직접 드로잉 + PDF 재렌더링 QA` 로 고정한다.
- `감정평가서(COL-02)`도 동일 원칙으로 표지, 감정평가표, 산출근거, 위치도, 사진용지를 포함한 9p 대표본을 제작해 workbench에 적재했다.

## 감정평가서류 지도/사진/펀칭 처리 원칙
- 사진은 법적 리스크를 낮추기 위해 실제 사진 수집본보다 생성형 이미지 기반 asset pool을 우선 사용한다. 사용자가 별도로 승인한 실제 사진 pool은 후순위 확장 옵션으로 둔다.
- 지도는 직접 그린 가상 지도를 기본값으로 두지 않고, 라이선스와 attribution을 명시한 로컬 지도 asset pool을 사용한다.
- OpenStreetMap 기반 asset은 이미지 내부 attribution과 별도 metadata JSON을 유지하고, 대량 생성 전에는 자체 호스팅 또는 사용 허가가 명확한 공급자로 전환한다.
- 펀칭/각인형 일련번호는 `datafactory.rendering.punched` 모듈을 보존하되, 자연스러움이 충분히 검증될 때까지 renderer 기본값에서는 비활성화한다.
