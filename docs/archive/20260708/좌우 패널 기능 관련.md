# 좌측 패널

## 1. 문서 목록

### 문서 필터링 기능
- 도메인별: ms 엑셀의 필터 기능처럼, 분야 다중 선택 기능이 추가되면 좋을듯.
- 상태별: 현재 상태가 매우 세분화되어있어 복잡하므로, 이것을 미착수/진행중/완료 3가지로 축약하는 작업이 필요할듯. 또한 여기에도 ms 텍셀처럼 다중 선택이 들어가면 좋을듯.
- 문서 검색: 현재 상태로도 충분하지만 정규화까지 지원하면 훨씬 도움될듯.

### n차 목표 문서
- 현재는 1차 목표 문서를 하드코딩하여 수동으로 그룹을 만들어둔 상태, 이를 일반화하여 지정한 그룹에 대해서 n개의 합성 샘플(pipeline ready)+클린룸 샘플을 일괄로 뽑아내는 기능은 유용할듯.
- 그룹 관리 및 활성화 기능이 추가되면 충분히 일반화 가능할듯. 다만 사용성이 간편해야 할듯.

## 2. 자동 적재 및 적재현황 관리 기능
- 현상 유지해도 문제 없을듯.

# 우측 패널

## 1. seed sample 관리기능
- 있으면 좋긴 한데, 지금보다 ui가 차지하는 공간을 줄여볼 수 있을듯(버튼 2개를 한 행에 놓는다던가, 등).
- 기능 실행 시 완벽하게 문서에 적재되었던 샘플이 들어내지고, 문서가 샘플 미적재 상태로 되돌아가야 함.

## 2. pipeline 관련 기능 패널들

### bbox 검출
- 현상 유지, 다만 ui가 차지하는 영역을 줄이고 버튼 텍스트를 좀더 단순화할 수 있을듯.
- 리뷰 재생성 버튼과 관련 기능은 들어내고, 단순하게 ocr 검출 시에 bbox 리뷰 초안이 덮어씌워지도록 하면 될듯(다만, 그 전에 백업 생성).

### bbox 리뷰
- 기능 현상 유지, 단 용어가 헷갈리기 때문에 더 직관적으로 변경(치황 > 사용, 보존 > 미사용). "무시" 유형은 없애고, 대신 선택 bbox를 삭제할 수 있도록 함.
- ui가 차지하는 영역 축소.

### 인페인팅
- 4-in-1 관련 기능 및 버튼 전체 들어내기.
- ui 영역 축소.

### 클린업
- 전면 재설계가 필요한 영역.
- 실질적으로 사용해 본 결과, 잡티가 남은 영역을 자유 올가미 마스크 선택 후 후처리 인페인팅을 해도 별반 달라지는 건 없었음.
- 따라서, 그냥 스포이드로 주뱐 섹상을 집어내서 직접 지우개처럼 지우는 그림판 같은 도구로 변형하는 게 더 효과적일 듯.
- 관련하여 더욱 스마트한 방법이 있다면 제안 바람.
- 현재 워크벤치에 있는 클린업 템플릿을 사용하고 있는 문서 스코프가 있다면 해당 템플릿을 인페인팅 템플릿으로 사용할 수 있도록 파일 이동 및 파일명 변경을 수행하고, 클린업 마스크 등의 관련 데이터 청소 필요.

### 합성 1-cycle
- 현재 1매 합성의 경우 실질적으로 사용하고 있지 않으므로, 본 섹션을 스키마 편집/스타일 부여/faker 프로파일 3게의 기능으로 쪼개서 각각 네이밍한다.
  - 스키마 섹션
    - 스키마 초안을 만들기 위해서는 bbox 정보와 함께 ai-agentic하게 동작하는 명령이 필요하다.
    - 목표로 하는 스키마 초안의 조건은 다음과 같다:
      - 메타데이터 없이 심플하게 kie 관점에서의 key-value 구조(value = "" 고정).
      - ocr 텍스트와 문서 이미지의 시각 정보를 병행 사용하여 실제 문서에서 확인 가능한 텍스트를 사용한 hierarchy를 갖는 key들의 구조를 json으로 구축한다.
      - 문서에서 텍스트로 직접 드러나지 않는 추상적인 구조체를 사용한 계층화 작업은 금물이다.
      - key_name은 협업자 누구나 쉽게 이해할 수 있도록 문서에 드러난 텍스트를 기반으로 가급적 한국어 자연어 형태로 구축한다(예: "고객 성명", "계좌번호" 등등).
      - 현재 별도의 api key 가 없으므로 별도의 codex agent를 호출하여, 지정된 프롬프트와 문서 이미지, ocr, bbox review 데이터를 입력값으로 전닿나다.
    - 반환된 스키마 초안은 raw json edit 방식으로 사용자가 편집할 수 있다.
    - 이렇게 만들어진 스키마는 기존의 semantic schema에 해당하며, 이를 사용하여 전체 매핑 정보 등을 담고 있는 전체 스키마를 만들 수 있어야 한다.
    
  - 스타일 부여 섹션
    - 스타일시트는 bbox 의존 형태의 데이터이므로, bbox 리뷰 데이터가 만들어지는 순간 자동적으로 스타일시트 초안이 생성되도록 한다.
    - 현재 구현되어 있는 authoring > style 편집 기능을 그대로 들고오되, 폰트의 경우는 기존처럼 가능한 모든 폰트를 싹 다 긁어오기보다는 '한국어가 지원되는' 폰트를 불러온다.
    - 가능하다면, 폰트 선택 시 해당 폰트의 미리보기가 제공되면 좋을 듯.
    - 스타일+스키마+faker 프로파일 3가지가 모두 (초인이더라도) 준비되면 문서 선택 시 즉시 렌더링 preview가 보여지도록 지원한다(현재는 authoring 불러오기 버튼을 클릭해야 해서 귀찮음).

  - faker 프로파일 섹션
    - 데이터 차원에서의 작업 및 기능 자동화 수준 개선의 작업 양 쪽이 모두 이뤄져야 하는 상태.
    - 데이터 차원:
      - 현재 1차 범위 문서들에 대해서 만들어져 있는 profile 들을 유형화하여 통합할 수 있는 것은 통합하며, 사용자가 편리하게 '다시 꺼내서 사용'할 수 있도록 준비하여야 함.
      - value data pool을 직접 구축한 것들도 가능하다면 합칠 수 있는 것들끼리 합쳐서 덩치를 키워 다양성을 높이고, 간단한 검색 등을 통해서 원하는 내용을 라이브러리에서 꺼내 쓰듯이 사용할 수 있어야 함.
    - 기능 자동화 차원:
      - 전에 없던 새로운 profile을 제작할 별도의 방법이 없음. 따라서, 문서 이미지+대응되는 schema+user description(직접 입력)을 별도의 codex agent에게 지정된 프롬프트 등과 함께 전달하여, 새로운 profile(+value data pool)을 제작하여 라이브러리에 추가하여 사용할 수 있도록 하는 반자동화 기능이 필요.
      - 가장 어려운 부분 중 하나라 추가 방법론 연구 및 논의 필요.

### 문서 합성 렌더링 섹션
- 현재처럼 테스트를 위한 단건 렌더링, 단건문서 n매 렌더링 기능을 남김. 전체문서 배치 렌더링의 경우 좌측 패널의 'n차 목표 전체 렌더링' 기능으로 이관.
- ui 영역 축소.
