# 수기체 데이터셋 생성 마스터 계획

작성일: 2026-07-08  
대상 프로젝트: DataFactory  
상태: 좌우 패널 기능 개발/개선 완료 이후의 신규 마스터 계획

## 1. 목적

레지스트리의 `작성방식=수기` 문서는 인쇄체 텍스트 렌더링으로 합성하지 않는다. 대신 기계가 정답값과 GT를 먼저 만들고, 사람이 그 값을 보고 실제 손글씨로 작성한 스캔본을 사전 생성 GT와 자동 매칭하여 데이터셋을 구축한다.

## 2. 핵심 원칙

- 손글씨 이미지는 AI/폰트로 합성하지 않는다.
- schema/faker profile/value set/GT label은 기계가 사전에 생성한다.
- 실제 필기체는 작업자가 답안지를 보고 빈 템플릿 인쇄본에 직접 작성한다.
- 스캔본과 GT의 매칭은 QR/barcode를 통해 기계적으로 수행한다.
- QR/barcode 영역은 최종 이미지에서 제거 또는 인페인팅한다.
- 최종 데이터셋은 `handwritten scan image + pre-generated GT label` 페어로 구성한다.

## 3. 수기체 생성 워크플로우

```text
템플릿 + bbox 준비
  -> schema 정의
  -> faker profile 정의
  -> faker value set 생성
  -> 스타일 없는 답안지 생성
  -> GT label 사전 생성
  -> 식별 QR/barcode가 삽입된 빈 템플릿 생성
  -> 답안지 + QR 템플릿 1세트 인쇄
  -> 작업자가 답안지 값을 보고 손글씨 작성
  -> 작성된 템플릿 인쇄본 스캔
  -> QR/barcode decode로 GT 자동 매칭
  -> QR/barcode 영역 제거 또는 inpaint
  -> 최종 이미지 + GT label 페어 확정
```

## 4. 산출물 구조 제안

문서별 workbench 하위에 수기체 전용 run을 둔다.

```text
workbench/documents/{문서명}__{doc_id}/handwriting_pipeline/
  print_packs/{run_id}/
    answer_sheets/
    qr_templates/
    values/
    gt/
    manifest.json
  scanned_intake/{run_id}/
    raw_scans/
    decoded/
    qr_removed/
    matched_gt/
    review/
    manifest.json
```

## 5. Print pack

Print pack은 실제 인쇄 전에 만들어지는 작업 묶음이다.

필수 구성:

- `answer_sheet`: 작업자가 보고 베껴 쓸 값 목록
- `qr_template`: QR/barcode가 삽입된 빈 템플릿 이미지 또는 PDF
- `values`: faker value set
- `gt`: 사전 생성 semantic/full GT label
- `manifest`: sample id, doc id, QR payload, GT path, 템플릿 path, 생성 시각

QR payload는 최소한 다음을 포함한다.

```json
{
  "project": "datafactory",
  "doc_id": "...",
  "sample_id": "...",
  "run_id": "...",
  "gt_path": "..."
}
```

## 6. Scan intake

스캔본 적재 시 다음 순서로 처리한다.

1. 이미지/PDF 스캔본 입력
2. QR/barcode 검출 및 decode
3. decode payload로 print pack manifest 조회
4. 대응 GT label 자동 연결
5. QR/barcode 영역 제거/inpaint
6. 최종 이미지와 GT 페어 생성
7. decode 실패, 중복, GT 누락, QR 제거 실패는 review_required 처리

## 7. SamsungLife에서 차용할 개념

`~/projects/SamsungLife`의 다음 개념을 차용한다.

- scanned/reference intake
- manifest 중심 lineage
- `pending / review_required / accepted / rejected` 리뷰 상태
- accepted-only export
- QR 검출/제거 후처리
- source kind 구분

DataFactory에서는 다음 source kind를 우선 사용한다.

- `synthetic_rendered`: 인쇄체 이미지 렌더 합성
- `editable_docx_rendered`: 보류/실험 상태. DOCX 값 주입 후 LibreOffice CLI 렌더를 검토할 수는 있으나, 폰트 재현 품질이 해결되기 전까지 주 합성 파이프라인과 최종 자동 산출 대상으로 사용하지 않는다.
- `handwriting_print_pack`: 수기 작성을 위한 인쇄팩
- `handwriting_scanned`: 작업자 필기 후 스캔본
- `handwriting_qr_removed`: QR 제거 완료본
- `human_reviewed`: 사람이 검수 승인한 최종본

## 8. 작성방식 정책

- `작성방식=인쇄`: 기존 이미지/PDF authoring을 기본 경로로 사용한다. DOCX 전용 파이프라인은 폐기/보류 상태이며, 후일 LibreOffice의 서식 보존과 프로젝트 보유 공문서형 폰트 일원화 전략이 검증될 때만 재검토한다.
- `작성방식=수기`: 최종 이미지 텍스트 렌더링 금지, print pack + scan intake 흐름 사용
- 값 생성, schema/faker profile 정의, GT 생성은 인쇄/수기 모두 동일한 semantic schema 기준을 유지한다.
- 사용자 primary schema는 semantic schema이며, full schema/binding은 시스템 lineage와 anchor 매핑용이다.

## 9. 구현 단계

1. 레지스트리 `writingMethod`를 UI에 명확히 표시한다.
2. 수기 문서에 대해 일반 렌더 최종 산출물 생성을 차단하고 print pack 생성 대상으로 분기한다.
3. QR/barcode 삽입 템플릿 생성 API를 추가한다.
4. 답안지 PDF/이미지 생성 API를 추가한다.
5. 스캔본 intake와 QR decode 매칭 API를 추가한다.
6. QR 제거/inpaint 후처리와 review 상태를 추가한다.
7. accepted-only export에서 수기 스캔본 + 사전 GT를 최종 results에 포함한다.

## 10. 미결정 사항

- QR/barcode 삽입 위치 기본값과 문서별 override 방식
- 문서당 기본 수기 샘플 수
- 작업자별 handwriting diversity metadata를 남길지 여부
- QR 제거 방식: 단순 주변색 fill, OpenCV inpaint, LaMa 중 기본값
- 스캔본 skew/rotation 보정의 자동화 범위
