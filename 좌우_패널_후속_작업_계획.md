# 좌우 패널 후속 작업 목표 및 실행 계획

작성일: 2026-07-07
기준 문서: `좌우 패널 기능 관련.md`
현재 기준: 패널 정리 1차분(필터/상태그룹/seed 되돌리기/BBox 용어/브러시 보정/Authoring 자동 로드)은 구현 완료 상태.

## 0. 공통 원칙

1. 기존 산출물은 삭제하지 않고, 마이그레이션 또는 되돌리기 작업 전에는 백업/보관 위치를 명시한다.
2. 렌더러는 authoring 데이터를 임의 보정하지 않고, authoring/schema/style/faker에 기록된 값을 그대로 따른다.
3. UI 정리는 기능 축소가 아니라 사용자가 실제로 실행하는 흐름을 기준으로 재배치한다.
4. 각 단계는 작은 단위로 구현하고, `py_compile`, `pytest`, `web npm build`로 검증한다.
5. 큰 데이터 마이그레이션은 별도 dry-run/리포트 모드를 먼저 만든 뒤 적용한다.

## 5. Authoring: Schema / Style / Faker / Render 분리

### 목표
합성 1-cycle 패널을 실제 작업 단위인 Schema, Style, Faker, Render로 나누고, 각각의 자동화 수준을 높인다. 또한 PO가 선호하는 최종 산출 형태가 COM 방식의 Office 문서인 점을 반영하여, PDF/JPG 기반 샘플뿐 아니라 DOCX 등 편집 가능한 빈 템플릿 문서 양식을 authoring 입력으로 사용할 수 있게 한다.

### 편집 가능한 템플릿 문서 양식 / Office COM 경로
- [ ] DOCX 형태의 “편집 가능한 템플릿 문서 양식”을 seed/sample 유형으로 적재할 수 있게 한다.
- [ ] 자유 양식 문서나 산문형 문서처럼 이미지 기반 bbox 수작업이 비효율적인 경우, 빈 DOCX form에 faker 값을 채운 뒤 Office COM 또는 호환 렌더러로 PDF를 생성하고, 그 렌더 결과에서 bbox/label/GT를 산출하는 경로를 설계한다.
- [ ] DOCX 템플릿 입력은 원본 템플릿, 채워진 DOCX, 렌더링 PDF, 페이지 이미지, bbox/label/GT 사이의 lineage를 manifest에 남긴다.
- [ ] 로컬/서버 환경 차이를 고려해 COM Office 렌더링은 별도 실행 backend로 추상화하고, 비-Windows 또는 Office 미설치 환경에서는 “렌더 대기/외부 렌더 필요” 상태로 표시한다.
- [ ] PDF/JPG 샘플과 DOCX 템플릿 샘플을 동일한 문서 상태 체계에서 다루되, 생성 경로는 `image-template`과 `editable-office-template`로 분리한다.

### 빈 템플릿 기반 agentic schema/faker 생성 규칙
- [ ] 빈 PDF/JPG/DOCX 템플릿을 입력으로 schema/faker profile을 agentic하게 생성할 때의 프롬프트 규칙과 금지사항을 명문화한다.
- [ ] Schema 초안은 KIE 관점 key-value hierarchy만 허용하고, 모든 value는 빈 문자열로 둔다. key는 문서에 실제로 보이는 라벨/표제/placeholder/주변 텍스트 기반 한국어 자연어를 우선한다.
- [ ] 문서에 보이지 않는 추상 키, 업무 추론만으로 만든 키, downstream 편의를 위한 임의 구조체는 금지한다. 필요하면 `needs_user_decision` 또는 `uncertain_fields`로 별도 보고한다.
- [ ] Faker profile은 schema key별 의미가 충분히 명확한 경우에만 faker rule을 제안한다. 의미가 불확실한 key는 `literal:`, `choice:`, `pool:` 등을 임의 생성하지 말고 보류 사유를 남긴다.
- [ ] Agent는 schema/faker profile 초안 생성 전에 실제 해당 문서명을 웹 검색하여 작성 방법, 법적/업무적 의미, 문서가 담는 정보 항목, 실제 샘플 양식, 기관/기업/공공 출처의 설명을 수집하고, 그 근거를 종합 분석한 뒤 schema와 faker profile 후보를 생성한다. 특히 faker profile은 문서 필드의 의미와 실제 작성 관행을 근거로 타입, 형식, 값 범위, 선택지, 단위, 날짜/금액/식별번호 규칙을 제안한다.
- [ ] 웹 검색 근거는 출처 URL, 검색일, 요약, schema/faker에 반영된 필드 단위 근거를 `research_report`로 남긴다. 공식/원문/실제 샘플 출처를 우선하고, 출처 간 충돌이나 불확실성이 있으면 faker rule을 확정하지 않고 `uncertainty_report`에 남긴다.
- [ ] 웹 검색 결과는 문서에 없는 필드를 새로 만들기 위한 권한이 아니라, 문서에 보이는 anchor의 의미 해석과 faker profile 정밀화를 위한 보조 근거로만 사용한다. 검색으로 알게 된 일반 항목이라도 템플릿에 대응 anchor가 없으면 schema에 자동 추가하지 않는다.
- [ ] 빈 템플릿 기반 생성에서는 실제 개인정보/실제 기업정보를 만들지 않고, 합성 가능한 더미 값 규칙 또는 외부 승인된 value pool 참조만 사용한다.
- [ ] DOCX 템플릿의 경우 visible text, content control, form field, table cell, bookmark/placeholder 등 “편집 가능한 anchor”를 추출 근거로 삼되, 숨은 메타데이터나 파일명만으로 key를 만들지 않는다.
- [ ] Agent 산출물은 바로 적용하지 않고 `schema_draft`, `faker_profile_draft`, `value_pool_draft`, `uncertainty_report`로 저장한 뒤 UI에서 승인/수정/백업 후 확정한다.

### Schema
- [x] 문서 이미지 + OCR + bbox review를 입력으로 agentic schema 초안 생성 기능을 설계한다. 로컬 규칙 기반 생성은 제외하고 agent 요청 패키지를 생성한다.
- [ ] 초안 조건: KIE 관점 key-value hierarchy, value는 빈 문자열, 문서에 드러난 텍스트 기반 한국어 key.
- [ ] raw JSON 편집 UI를 별도 schema 섹션으로 분리한다.
- [ ] semantic schema를 전체 authoring schema로 변환하는 단계를 만든다.
- [ ] 빈 템플릿 입력(DOCX/PDF/JPG)에서 추출한 anchor와 schema key를 연결하는 중간 representation을 정의한다.

### Style
- [x] 한국어/CJK 후보 중심 폰트 필터링.
- [x] style 편집 기능 유지.

- [ ] bbox review 생성 순간 stylesheet 초안을 자동 생성하는 안전한 방식 설계.
- [ ] 폰트 선택 미리보기 제공.
- [ ] style 섹션을 UI상 더 명확히 분리한다.

### Faker
- [ ] 기존 profile을 유형화/통합한다.
- [ ] value data pool을 통합하고 검색/재사용할 수 있게 한다.
- [ ] 문서 이미지 + schema + 사용자 설명을 바탕으로 새 profile/value pool을 agent가 제안하는 흐름을 설계한다.
- [ ] 생성 결과를 라이브러리에 추가하는 승인/백업 흐름을 만든다.

### Render

- [x] 전체/그룹 배치 렌더링은 좌측 목표 그룹 영역으로 이관한다.
- [ ] 렌더 결과 링크와 검증 리포트를 compact하게 표시한다.
- [ ] DOCX 템플릿 기반 렌더링은 채워진 DOCX와 PDF 변환 결과를 함께 표시하고, bbox/label 생성 결과의 근거 파일을 추적 가능하게 한다.

## 6. 검증/완료 기준

각 구현 단계는 다음을 만족해야 완료로 본다.

- Python 변경: `./.venv/bin/python -m py_compile ...`
- 테스트: `env PYTHONPATH=src ./.venv/bin/python -m pytest -q`
- Web 변경: `cd web && npm run build`
- 데이터 변경 또는 마이그레이션: dry-run 리포트와 백업 위치 확인
- 최종 보고: 변경 파일, 완료 항목, 미완료/위험 항목 명시
